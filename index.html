The problem we're solving

Bottlenecks in
Massport's lease operations

The current Kelar CLMS has served Massport for years. But it was never designed for the scale, diversity, or compliance obligations of today's portfolio. These are the critical gaps that a modern ALMS directly addresses.

Not user-
friendly

Usability gap across user groups

Most users outside the core Aviation team do not consider the system user-friendly. The system was built for the Aviation department and has not served the Real Estate group or other users as well. It also lacks user-defined fields — any new field requires a database and code change rather than in-app configuration — and cannot present different layouts per business line.

Manual
billing

No direct financial system integration

Finance downloads the monthly billing file from Kelar and manually uploads it to the financial system. There is no direct integration. The move to Oracle Fusion makes this an immediate priority to address.

Static

Reporting mechanism

Currently Crystal Reports are used, which is static in nature. It doesn't allow a good level of flexibility that is needed for different stakeholders.

No GIS
for RE

Real Estate group lacks spatial capability

The Real Estate group does not have GIS capability within the current system. They also don't have anyone on their team to maintain it — a gap the new platform needs to accommodate with a low-maintenance option.

No
mobile

No mobile or handheld access

The Real Estate group has specifically requested mobile access. For Aviation staff walking terminals and reviewing spaces with potential tenants, handheld access to lease and floor plan data would be a clear benefit.

Stop-
gap

Current upgrade is only a holdover

Kelar is performing an upgrade right now — but only to extend the system's life for a few more years. Massport's stated intent is full replacement. The system's high customization has made finding a commercial substitute difficult. Confirmed during discovery: the current system runs on-premises on Adobe ColdFusion and Tomcat; the replacement is to be deployed on cloud.

Proposed Architecture

Built to grow with you.
Never to lock you in.

Every module in the ALMS is a pluggable component. The MVP and Phase 1 deliver a fully operational system. Extended modules connect via internal APIs and can be included in any phase — or deferred indefinitely — with no re-architecture of the core.

Plugin principle: The core platform exposes internal API endpoints for every extended module from day one. Modules you don't select are simply not deployed — no code is removed, no redesign is required. Any module can be added via a future contract amendment with no disruption to the running system.

Proposed System integrations

Connects to what
Massport already runs

The ALMS integrates with Massport's existing technology stack. Mandatory integrations are delivered in MVP and Phase 1. Optional integrations activate automatically with their respective plugin modules — no additional integration work is required when a module is selected.

All integrations use REST APIs where available, with SFTP file exchange as a fallback.

All data in transit encrypted with TLS 1.3 minimum.

All data hosted in US-based cloud region to comply with TSA data sovereignty requirements.
Oracle Fusion ERPBi-directional REST API (AR + Procurement/CLM)
Invoices pushed to Oracle AR on billing run completion. Payment status returned to ALMS. Automated GL journal entries for revenue recognition and deferred revenue. Executed lease agreements and structured terms received from Oracle Fusion's procurement/CLM module once Massport consolidates contract origination there — see Module detail.

→ Massport DTIG / System Admin
